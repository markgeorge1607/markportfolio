The Problem

Research
Core problem
Shopify payouts are a black box

Shopify pays merchants in lump sums — a single deposit that bundles gross sales, refunds, transaction fees, gateway charges, and chargebacks into one number. That number almost never matches what the merchant sees in Xero or QuickBooks. The delta is the problem.

Standard integration tools — like Shopify's native Xero connector or third-party apps like A2X and Link My Books — talk to your accountant. They move data between systems. They don't explain anything. A merchant still has to open a spreadsheet, cross-reference three different reports, and manually hunt for the line that doesn't reconcile. For most small business owners, that's hours every month.

Research signal
What merchants actually say

"Every month I spend two or three hours trying to work out why my Shopify payout doesn't match what landed in the bank. My accountant can't figure it out either. We just write it off as 'reconciling items' and move on."

QuickBooks UK Community Forum

"The Xero integration just dumps a lump sum. It doesn't tell you it's 43 orders minus a PayPal reserve minus two refunds from October. You find that out yourself at 11pm."

Shopify Community · Payout discrepancy thread

"I've tried three different reconciliation apps. They all sync the data. None of them explain it. The discrepancy still exists — I just have cleaner books with the same hole in them."

Shopify App Store review · Reconciliation category
Key insight from community research: The core problem isn't data transfer — it's comprehension. Merchants don't need better sync tools. They need someone to tell them, in plain English, exactly what happened to their money. That's the gap Klaio fills.
Competitive landscape
Where Klaio sits
A2X / Link My Books
Accountant-first tools
Move data cleanly between Shopify and Xero. Built for accountants who understand the structure. Merchants still can't read the output without help.
Klaio.io
Merchant-first intelligence
Reconciles every payout automatically and explains exactly what happened — in one sentence — without requiring any accounting knowledge. Built for the merchant, not their accountant.

Concept & Pivot

Complete
Origin
Started as something broader

The first version of the product — called Coreflow — was a unified SMB intelligence dashboard connecting Shopify, HubSpot, and QuickBooks into a single real-time view. The concept was compelling: see every customer event, from first purchase to invoice, in one place. The problem was scope.

Three OAuth integrations, three webhook streams, fuzzy identity matching across platforms — for a solo founder using Claude Code, that's a significant surface area before a single paying customer exists. After the initial architecture and prototype were mapped out, the decision was made to narrow.

The pivot decision
Shopify payout reconciliation as the beachhead

Community research across Shopify forums, QuickBooks UK support threads, and Xero user groups surfaced a specific, painful, recurring problem — payout discrepancies — that existing tools weren't solving from the merchant's perspective. That was the pivot point.

Decision rationale: One integration (Shopify → Xero/QuickBooks) is shippable. Three integrations is a roadmap. The payout problem is specific enough to solve well, frequent enough to retain on, and painful enough that merchants will pay to fix it. Start here, expand later.
Consideration Coreflow (original) Klaio (pivot)
Integration count 3 platforms 1 platform → 1 accounting tool
Time to first value Weeks of infra setup 15-minute onboarding
Target user clarity Any SMB with 3 tools Shopify merchants using Xero or QuickBooks
Problem specificity General visibility gap Payout reconciliation — specific pain, known frequency
Solo build feasibility High complexity Achievable in 10–14 weeks with Claude Code

Brand & Design System

Complete
Naming
From Coreflow to Klaio.io

Coreflow had naming conflicts. The rename wasn't just a domain decision — it was an opportunity to give the product a more distinctive identity. Klaio.io was available, short, and pronounceable. The .io suffix signals a developer-adjacent, modern SaaS product. The wordmark carries it.

Logo
Full wordmark · Space Grotesk 700
Klaio.io

The logo is a pure wordmark — no icon. Deliberately chosen to avoid the templated compass/arrow marks that crowd the fintech and SaaS space. Space Grotesk at 700 weight gives it confidence. The .io suffix drops to 400 weight at 60% opacity — present but subordinate, so it doesn't compete with the name. Primary context is white-on-green. Secondary is green-on-parchment.

Colour system
Green-forward, with restraint

The palette is anchored in a deep forest green — warm and serious without feeling clinical. Parchment as the background avoids the cold-white look of most fintech products. Status colours (amber, red) are muted versions of their defaults to stay within the brand register.

Typography
Space Grotesk + DM Mono
Display + Body
Space Grotesk
Used for all headings, body copy, and UI text. 700 for display, 400 for body. The tight letter-spacing (-0.05em at large sizes) gives it density without feeling heavy.
Mono — Labels, tags, metadata
DM Mono
Used exclusively for eyebrows, status tags, monospaced values, and structural labels. Creates a technical register without using a generic coding font.
Icon system
Iconoir — exclusively

One icon library, used throughout with no exceptions. Iconoir's line style sits cleanly against the Space Grotesk wordmarks without competing. All icon usage via CDN class names — no SVG embeds, no custom icons.

Design system
Token-first, Claude Code-enforced

The design system was built as a living HTML reference document — klaio-design-system.html — containing every CSS custom property, component, button variant, pill style, card type, and surface pattern used across the product. It doubles as the source of truth and a copy-paste library for Claude Code.

All tokens are defined in a single :root block. No hardcoded hex values appear outside of it. This makes the entire visual system changeable from one file.

Claude Code enforcement: A CLAUDE.md file at the project root instructs Claude Code to check the design system before any UI work. The Pre-Build Decision Protocol mandates: check for an existing component first, use only --dt-*-equivalent tokens, never add custom CSS to existing components. This prevents design drift across long build sessions.

Goal
Validate the problem before building the product

The landing page isn't a marketing site for a finished product. It's a validation instrument. The job is to answer one question: do Shopify merchants with a payout reconciliation problem care enough to give us their email?

The design decision follows from that: no pricing, no feature list, no customer logos. Just the problem stated clearly, the solution described concisely, and a waitlist form with as little friction as possible.

Design decisions
Every choice is intentional
Hero — Green full-bleed
Anchor the brand immediately
The hero section is the only place in the product world where the deep green fills the entire viewport. It signals confidence and makes the white wordmark unmistakeable. No stock imagery, no illustration — the grid texture does the work.
Headline — Two-line structure
"Your Shopify payout, explained in plain English."
The first line (bold) names what the merchant already knows — their payout. The second line (light weight, reduced opacity) states what Klaio does. The typographic contrast makes the proposition scannable in under 2 seconds.
CTAs removed
Try free / Book a demo → Waitlist only
Original CTAs implied a working product. Replaced with a single name + email waitlist form. "Join waitlist" sets correct expectations. "No spam. No credit card. Just an early invite." removes the remaining hesitation.
Sections removed
Pricing and Customers stripped out
Pricing implies a ready product and invites price objections. Fake customer logos undermine trust. Both were removed. The page earns credibility through problem clarity, not social proof theatre.
App UI preview
Show the real product, even before it exists

Below the fold, a browser-chrome mockup shows the actual dashboard design — payout list, reconciliation stats, attention-needed alerts, AI summary panel. This isn't decorative. It anchors the merchant's imagination to something concrete and shows the product's north star interface. The mockup uses real Klaio design system components, so it's consistent with what will ship.

Technical setup

Static HTML, no framework dependency. Fast, free to host, zero infrastructure to maintain during the waitlist phase. Formspree handles email capture and forwards signups without needing a backend. The site is intentionally over-engineered only where it matters: the visual design.

Product Architecture

Designed
Technical stack
Chosen for solo-founder build speed
Next.js 14 TypeScript Supabase Prisma BullMQ Redis Vercel Railway

Every stack decision was made with solo feasibility in mind. Supabase handles auth and RLS without custom server code. Prisma gives type-safe database access. BullMQ handles the async reconciliation jobs that run after each Shopify payout webhook. Railway hosts the worker process separately from the Next.js app on Vercel.

Gateway adapter pattern
Three tiers of integration, matched to feasibility

Not all payment gateways expose the same data. The gateway adapter pattern handles this without building three entirely different reconciliation engines.

T1
Shopify Payments
Full automation. The Shopify Balance Transactions GraphQL API exposes every fee line, refund, and chargeback at the payout level. Klaio fetches, normalises, and reconciles without any merchant input.
Full auto
T2
PayPal & Stripe
API with fuzzy matching. Payouts are fetched via their respective APIs, then matched against Xero bank feed entries using amount + date proximity. Exceptions flagged for merchant review.
API + review
T3
Klarna & others
CSV upload. Gateways without usable APIs. Merchant uploads their payout export, Klaio parses it and runs the same reconciliation logic. Less seamless, but complete coverage.
CSV upload
AI summaries
Claude explains every payout in one sentence

After reconciliation, Klaio calls the Claude API with a structured payout summary — total, line-by-line breakdown, discrepancies — and receives a plain-English explanation. This is the core differentiator. Not a data dump, not a report: a sentence a merchant can read and understand without accounting knowledge.

Example AI summary output
"Your April 25 payout of £3,140.20 covered 28 orders. It reconciles cleanly against Xero — all 4 Shopify transaction fees (£34.80 total) are posted to your Fees account. No action needed."
When something's wrong
"Your April 28 payout of £4,541.50 is missing £106.20 in transaction fees from Xero — they were deducted by Shopify but never posted to your P&L. This overstates your profit by the same amount. Click to post the missing entry."

Onboarding
Six steps, 15 minutes to first insight

The onboarding flow was designed and prototyped before any backend was built. Six linear steps: create account → connect Xero (OAuth) → live chart of accounts fetch + account mapping → select gateways → connect Shopify → first reconciliation runs in background. The merchant sees their first report before they close the onboarding tab.

Key design decision — live chart of accounts: Instead of asking merchants to type account codes they don't know, Klaio calls the Xero API immediately after OAuth and fetches their real account list as dropdown options. Eliminates the most common onboarding failure mode.

Open Items & What's Next

Planned
Pre-launch blockers
What needs to happen before merchant data goes live
01
Next.js CVE — upgrade required
Current codebase is on Next.js 14.2.0 which has a documented security vulnerability. Shipping financial merchant data on a known CVE is not defensible under UK GDPR Article 32. Must upgrade before any live merchant connections.
Blocker
02
Xero token refresh — Redis lock
Xero access tokens expire every 30 minutes with rotating refresh tokens. Without a Redis distributed lock, two concurrent BullMQ jobs can race on token refresh, invalidate each other, and break the Xero connection. Fix is a SET key NX EX 30 lock per organisation.
Blocker
03
UK GDPR — Data Processing Agreement
Klaio is a data processor for each merchant. A GDPR-compliant DPA must be in place before any live merchant data is handled. This is a legal document, not a code change — needs drafting and including in the onboarding flow.
Pre-launch
04
Webhook HMAC verification — all gateways
Shopify, Stripe, and PayPal webhooks all include signature headers. These must be verified before processing. Without it, a bad actor could POST fabricated payout data to the endpoint and corrupt reconciliation records.
Pre-launch
05
Stripe billing — post-waitlist validation
Payments are intentionally not implemented yet. After waitlist validation, Stripe will be added: subscription pricing tiers, post-payment webhook to activate accounts, customer self-serve portal. Estimated 1–2 days of Claude Code work.
Post-validation

Build method
Solo founder + Claude Code as engineering partner

Every line of code, every architectural decision, and every design implementation in this project has been produced by a solo founder using Claude as the primary engineering collaborator. The workflow: Claude Chat for architecture, strategy, and design decisions; Claude Code for implementation and iteration in the codebase.

The CLAUDE.md Pre-Build Decision Protocol exists specifically to maintain quality across long Claude Code sessions — preventing over-engineering, enforcing the design system, and ensuring each build decision is checked against existing components before new ones are created.

Timeline: Concept to live waitlist in under 8 weeks, solo, part-time. Full product build estimate: 10–14 weeks to Tier 1 beta with paying customers.
